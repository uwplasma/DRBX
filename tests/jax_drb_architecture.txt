Architecture summary for the FCI / EB blob path in this repo

1. Overall layering

The code is organized in four practical layers:

- geometry: defines the logical mesh, metric, magnetic-field geometry, and FCI trace maps
- native boundary/reconstruction: turns cell-centered fields into face values, wall values, normal derivatives, and boundary payloads
- native operators: consumes stencils and geometry to compute parallel/perpendicular derivatives, Poisson brackets, curvature terms, and Laplacian-like operators
- model RHS / test harness: assembles the drift-reduced Braginskii evolution, solves phi, advances state with RK4, and records diagnostics

The key idea is that geometry is separate from reconstruction, reconstruction is separate from operator algebra, and operator algebra is separate from the physics model assembly.

2. Geometry layer

The geometry module provides the data structures that every operator consumes:

- Grid1D: one logical axis with centers and faces
- CellCenteredGrid3D: x/y/z Grid1D objects packaged into a 3D cell-centered grid
- FciMaps3D: field-line tracing results and endpoint metadata
- FciGeometry3D: the full geometry bundle, including:
  - grid
  - maps
  - spacing
  - cell_metric / face_metric
  - cell_bfield / face_bfield

Geometry is the shared contract:
- field arrays are always shaped like geometry.shape = (nx, ny, nz)
- face arrays are shaped like (nx+1, ny, nz), (nx, ny+1, nz), or (nx, ny, nz+1)
- spacing and metric objects are precomputed and reused everywhere

In the shifted-torus EB blob test, geometry is built from a specialized shifted-torus geometry helper, then the magnetic field is slightly modified to produce the intended axis-regular lower-rho behavior.

3. Boundary representation layer

The boundary code in fci_boundaries.py distinguishes several kinds of payload:

- BoundaryFaceBC3D:
  - dense regular-grid face boundary conditions
  - separate x/y/z arrays for kind, value, and mask
  - supports Dirichlet, Neumann, normal-flux, and no-flux semantics
  - used for the regular coordinate faces

- CutWallGeometry3D / CutWallBC3D:
  - sparse cut-wall geometry and payloads for embedded boundary faces
  - used only when the geometry has actual cut-wall surfaces

- BoundaryFlux1D / BoundaryFlux3D:
  - typed containers for lower/upper fluxes on each axis

The philosophy is:
- regular coordinate faces use dense face grids and face-level BC payloads
- cut walls use separate wall geometry and sparse wall payloads
- the operator layer only needs the reconstructed fields and boundary payloads, not the geometry construction details

4. Coordinate face reconstruction

CoordinateFaceValueReconstructor3D is the regular coordinate-face reconstructor.

Inputs:
- a cell-centered 3D field
- FciGeometry3D
- periodic_axes: 3-tuple of booleans
- axis_regular_axes: 3-tuple of booleans

Behavior:
- interior face values are always centered averages
- periodic axes use wraparound averages at the two ends
- nonperiodic axes use boundary extrapolation from the adjacent cell centers
- the lower-x face can optionally use an axis-regular special treatment
- y and z remain standard unless explicitly periodic

Important design rule:
- periodic_axes and axis_regular_axes cannot both be True on the same axis
- axis_regular_axes is currently only meant for the lower x axis
- that lower-x special case exists because the lower-rho / lower-x topology is not a standard physical wall in the shifted-torus setup

The Neumann variant extrapolate_neumann() uses the same face reconstruction logic but adds normal-derivative corrections on nonperiodic boundary faces.

5. Coordinate normal-derivative construction

CoordinateNormalDerivativeConstructor3D builds second-order normal derivative weights for regular coordinate faces.

What it stores:
- dnormal_weights: shape (3, 2, 4)
- d2normal_weights: shape (3, 2, 4)

Interpretation:
- axis index 0/1/2 = x/y/z
- side index 0/1 = lower/upper face
- the last dimension multiplies wall value plus the first three interior cells

How it works:
- from_geometry() constructs the weights by solving a small Vandermonde system for each axis and side
- the derivative convention is outward normal, not inward coordinate derivative
- dnormal_from_wall_value() and d2normal_from_wall_value() evaluate the wall derivatives from:
  - the cell-centered field
  - the reconstructed wall value tuple (x, y, z)
  - the geometry
  - periodic_axes and axis_regular_axes

The lower-x special case:
- if axis_regular_axes[0] is True, the lower x wall value is replaced by the axis-regular reconstructed value instead of being treated like a normal boundary wall
- the upper x face remains a genuine boundary face
- this is why the x-axis logic is asymmetrical in the current design

6. Cut-wall reconstruction

The cut-wall constructors are the sparse analogs of the coordinate-face constructors:

- CutWallValueReconstructor3D:
  - gathers a stencil of neighboring cells for each wall face
  - applies precomputed interpolation weights
  - optionally adds a Neumann correction via bc_coeff

- CutWallNormalDerivativeConstructor3D:
  - stores neighbor indices and derivative weights per wall face
  - computes outward normal first and second derivatives on cut walls
  - is fully wall-face local; it does not depend on dense coordinate faces

These are used when the geometry contains actual embedded boundaries or cut-wall features.

7. Stencil layer

The code uses two main stencil families:

- LocalStencil1D:
  - one direction’s minus/center/plus values plus local spacing
  - used by derivative and pointwise operators

- LocalStencil3D:
  - x/y/z LocalStencil1D objects bundled together
  - used for pointwise operators like grad_parallel, poisson bracket, curvature, etc.

- ConservativeStencil3D:
  - same structural shape as LocalStencil3D
  - semantically restricted to conservative / flux-assembly operators
  - kept separate so conservative routines cannot accidentally inherit local embedded-wall semantics

There are also flux containers:
- FaceFluxStencil3D
- LocalControlVolumeFluxStencil3D

These are used in conservative divergence assembly.

8. Native operators

The operator layer consumes stencils and geometry to produce actual derivatives and PDE terms.

Main operators in fci_operators.py:

- grad_parallel_op_direct():
  - computes b^i partial_i f from a LocalStencil3D
  - uses finite differences on each axis and contracts with the contravariant magnetic field

- parallel_laplacian_direct_op():
  - applies grad_parallel twice, with stencil reconstruction in between
  - supports periodic axes and regular face BCs

- grad_perp_op():
  - projects a gradient perpendicular to the magnetic field using metric and B

- perp_laplacian_local_op():
  - local, pointwise approximation of the perpendicular Laplacian
  - uses face divergence of projected flux

- poisson_bracket_op():
  - computes the logical Poisson bracket using the magnetic geometry and finite differences

- curvature_op():
  - contracts a supplied curvature coefficient field with the local gradient

- perp_laplacian_conservative_op() and related helpers:
  - conservative flux-based perpendicular Laplacian
  - uses regular face fluxes, cut-wall contributions, and boundary conditions

A useful architectural split here:
- direct operators work on stencils and local reconstruction
- conservative operators work on fluxes and divergence balance
- boundary and cut-wall handling is pulled up into the flux assembly stage, not buried inside the divergence kernel

9. Boundary-conditioned flux assembly

The conservative operator path builds a boundary-complete flux stencil before taking divergence.

That path:
- reconstructs gradients on faces
- patches face gradients at lower/upper boundaries using BoundaryFaceBC3D
- applies face flux BCs for normal-flux / no-flux / Dirichlet / Neumann semantics
- attaches cut-wall flux payloads if present
- returns a LocalControlVolumeFluxStencil3D

This keeps the actual divergence operation simple: it just balances fluxes on a control volume.

10. Drift-reduced Braginskii EB model layer

fci_drb_EB_rhs.py defines the model-level state and RHS assembly.

State:
- FciDrbEBState:
  - density
  - phi
  - Te
  - Ti
  - Vi
  - Ve
  - vorticity

Boundary-condition bundle:
- FciDrbEBBoundaryConditions:
  - per-field regular face BCs
  - per-field cut-wall BCs

Parameters:
- FciDrbEBRhsParameters:
  - normalization constants
  - diffusion coefficients
  - solver settings for phi inversion
  - parallel/perpendicular transport coefficients

RHS assembly:
- compute_fci_drb_eb_rhs() builds all field stencils
- it calls field-specific RHS helpers:
  - density
  - Te
  - Ti
  - Ve
  - Vi
  - vorticity
- each helper combines:
  - Poisson bracket
  - parallel advection / flux
  - curvature terms
  - parallel diffusion
  - perpendicular diffusion
- phi is not evolved directly; it is reconstructed by solving a perpendicular Laplacian constraint from vorticity and Ti

The model uses the operator layer as reusable physics primitives rather than re-deriving anything inside the RHS routines.

11. Phi reconstruction / inverse solve

The test harness uses a perpendicular Laplacian inverse solver to recover phi from the current state.

The flow is:
- build boundary conditions for the current state
- build a conservative stencil for Ti
- evaluate Ti’s perpendicular Laplacian
- form the phi RHS from vorticity and Ti coupling
- apply a preconditioned / multigrid-style inverse solve
- reconstruct phi and feed it back into the state before RHS evaluation

This makes phi a constrained variable rather than an independent time-integrated field.

12. Shifted-torus EB blob test architecture

test_shifted_torus_EB_blob.py is both a regression test and an integration harness.

Its structure is:

- build shifted-torus geometry
- impose the axis-regular lower-x special case
- initialize a Gaussian blob in density
- construct boundary conditions from the instantaneous state
- solve for phi
- advance the state with RK4
- record time histories
- save history files and diagnostic plots

This test is also where the axis-regular logic is exercised:
- PERIODIC_AXES = (False, True, True)
- AXIS_REGULAR_AXES = (True, False, False)

That means:
- x is nonperiodic but lower-x is handled with the axis-regular mapping
- y and z are periodic
- the lower-x face behaves like a topological axis rather than a standard physical wall

The test’s helper functions are doing three jobs:
- geometry assembly
- axis-regular stencil adjustment
- sheath-style boundary-condition construction for density, temperature, potential, and parallel velocities

13. How the test’s boundary conditions are built

The boundary-condition builder in the EB blob test:
- reconstructs face values for Te, Ti, phi, density, and Vi
- computes wall values for the sheath conditions
- derives outward normal derivatives on x faces
- uses those to form:
  - density Neumann BCs
  - temperature Neumann BCs
  - potential Dirichlet/Neumann coupling
  - vorticity Dirichlet BCs
  - Vi / Ve Dirichlet BCs

Only the outer x face is treated as a wall in this setup.
The lower x face is intentionally treated as axis-regular topology, so it is not used as a standard boundary wall.

14. Time integration and diagnostics

The time integrator is a hand-written RK4 loop:
- reconstruct phi at each stage
- compute RHS
- stage update
- reconstruct phi again
- repeat

Outputs:
- history arrays for each evolved field
- .npz history cache
- time-trace plots
- optional movie frames for density and phi

The test also includes diagnostics for the phi inverse solve:
- residual norms
- action checks against the expected operator split
- solved phi statistics

15. Architectural takeaways

The design is strongly layered and intentionally explicit:
- geometry is a first-class input, not hidden inside operators
- regular coordinate-face handling is separated from cut-wall handling
- local reconstruction is separated from conservative flux assembly
- direct derivative operators are separated from model physics
- phi is solved as a constrained elliptic subproblem rather than evolved directly

For the lower-x / axis-regular work specifically, the repo already treats that axis as special in the EB blob path:
- the test harness constructs axis-regular stencils and boundary payloads
- the coordinate face and normal-derivative constructors are the place where the lower-x asymmetry is enforced
- the model RHS simply consumes the resulting face values, derivatives, and boundary objects
